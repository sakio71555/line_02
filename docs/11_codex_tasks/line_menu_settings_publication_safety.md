# LINEメニュー設定の公開安全性

## 目的

管理画面で会社ごとのLINEメニューと質問フローを編集できる機能について、公開中の動作を壊さずに編集・追加・公開できる状態にする。

## 対応範囲

- 条件分岐は表示文ではなく変更されにくい内部値で判定し、既存の日本語ラベル保存値も互換変換する
- 新規メニューは下書きとして作成し、公開操作まで実行時フローから除外する
- 公開時の設定をスナップショットとして保持し、その後の編集を即時反映しない
- 公開済み・終了済みメニューの削除をAPIと画面の両方で禁止する
- 任意の追加メニューをLINE公開作業へ渡せる汎用JSON成果物とオペレーター入力を用意する
- 設定保存に更新日時を使った競合検知を追加し、他の担当者の更新を上書きしない

## 完了条件

- 新規メニューにダミーURLが入らず、保存だけではLINE応答に使われない
- 公開済みメニューを編集中も、お客様には直前の公開内容が使われる
- 条件付き質問が内部値・既存ラベル値のどちらでも正しく分岐する
- 同じ設定を別画面から先に更新した場合、古い画面の保存は409で停止する
- 公開済みメニューの削除要求は409で停止する
- 任意メニューの公開定義をダウンロードでき、既存オペレーターが任意アセットディレクトリを扱える
- lint / typecheck / test / integration test / build が成功する

## 追加メニューの公開手順

1. 管理画面でメニューと質問フローを下書き保存する
2. 「LINE公開用JSONを書き出す」で `rich-menu.json` を取得する
3. リポジトリ内の専用ディレクトリへ `rich-menu.json` と対応する `rich-menu.png` を配置する
4. 次のdry-runで定義と画像を検証する

```sh
npx tsx scripts/ops/line_rich_menu_operator.ts --asset-dir <repo-relative-asset-directory>
```

5. オペレーターの明示承認後に、同じディレクトリを指定して1回だけ公開する
6. 既定メニューへ切り替える場合に限り `--set-default` を明示する
7. LINE側で作成されたリッチメニューIDを管理画面へ設定し、「編集内容を公開」を押して会社設定を保存する

公開処理はLINEメッセージ送信を行わない。アクセストークン、リッチメニューID、顧客情報は標準出力やドキュメントへ記録しない。

## 対象外

- LINE Messaging APIへの実送信
- LINE上のリッチメニュー作成・切替の実行
- OpenAI API実行、DB直接接続、Nginx・DNS・証明書変更
