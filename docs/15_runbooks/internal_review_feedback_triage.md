# Internal Review Feedback Triage

## Purpose

このrunbookは、アマミホーム社内確認版で出たフィードバックを、その場で大きく実装せず、小さいCodex Loopへ変換するための運用ルールです。

社内確認では、UI、文言、実務フロー、AI下書き、LINE送信、本番ログイン、データ永続化などの意見が混ざります。すべてを一度に対応すると、tenant_id分離、外部API境界、未接続範囲、テストが崩れやすくなります。

## Audience

- アマミホーム社内確認の同席者
- フィードバックを記録する担当者
- Codexへ次Loopを依頼する開発者
- 本番化前の優先順位を決めるプロジェクト関係者

## When To Use

- 社内確認版の画面を見せた直後
- フィードバックを `docs/15_runbooks/internal_review_feedback_log.md` に転記するとき
- 似た意見をまとめて次Loop候補にするとき
- 「今すぐ直すもの」と「本番化で扱うもの」を分けるとき

## What To Record

フィードバックは、実装要望ではなく「観察された困りごと」として記録します。

- 受付日
- 確認者
- 画面/機能
- 内容
- 種別
- 影響度
- 優先度
- 実装難易度
- 社内確認版で対応するか
- 本番化で対応するか
- 判断
- 次Loop候補
- メモ

## Categories

| Category | Use When |
| --- | --- |
| UI / 文言 | 画面の見た目、ラベル、説明文、ボタン文言が分かりにくい |
| 業務フロー | 実際の相談対応、担当者引き継ぎ、確認順と画面の流れが合わない |
| AI下書き品質 | AI要約、返信下書き、注意点、次質問の表現を変えたい |
| RAG / 参考情報 | ホームページ情報、source、回答案の根拠、検索語に関する意見 |
| LINE返信 | 担当者返信、送信前確認、本物LINE送信に関する意見 |
| 未返信アラート | 未返信判定、alert表示、対応状況、重複防止に関する意見 |
| 通知 | 担当者通知、通知先、通知タイミング、schedulerに関する意見 |
| 顧客情報 | 顧客項目、タグ、電話番号、メール、希望条件などの不足 |
| 認証 / 権限 | ログイン、利用先選択、権限、管理者/担当者の操作差分 |
| データ永続化 | API再起動で消える、履歴保存、Supabase、本番DBに関する意見 |
| スマホ / LIFF | スマホ確認、LIFF相談フォーム、モバイル入力に関する意見 |
| 本番運用 | deploy、監視、ログ、バックアップ、運用手順、社内ルール |
| 不具合 | 起動不可、画面遷移不可、エラー、誤表示、期待と違う動作 |
| その他 | 上記に当てはまらないもの |

## Priority

優先度は「いつ扱うべきか」を表します。

| Priority | Meaning | Examples |
| --- | --- | --- |
| P0 | 社内確認が継続できない重大不具合 | 起動不可、主要画面が開けない、誤送信に見える |
| P1 | 社内確認の理解や操作に大きく影響するもの | 初心者が操作を誤解する、主要導線が分からない |
| P2 | 社内確認後、本番化前に対応したいもの | 本番送信確認、永続化、認証、通知タイミング |
| P3 | 将来改善でよいもの | 軽い見た目改善、便利な絞り込み |
| P4 | SaaS化や拡張時に検討するもの | 複数企業管理、請求、招待、監査ログ |

## Severity

影響度は「起きたときの重さ」を表します。

| Severity | Meaning |
| --- | --- |
| Critical | 起動不可、主要導線不能、誤送信リスク |
| High | 社内確認で誤解が大きい、実務判断に支障 |
| Medium | 使い勝手や文言の改善 |
| Low | 見た目や軽微な表現 |
| Info | アイデア、将来要望 |

## Effort

実装難易度は「どれくらいのLoopに分ける必要があるか」を表します。

| Effort | Meaning |
| --- | --- |
| S | 文言、docs、test程度 |
| M | 小さなUI変更、既存APIの範囲 |
| L | API、DB、認証、外部連携を伴う |
| XL | 本番運用設計、LIFF、SaaS化、複数tenant設計 |

## Review Edition / Production / SaaS Split

### 社内確認版で対応

- 画面理解
- 説明文
- demo flow
- 軽微なUI
- runbook
- 誤解を減らすバッジや注意書き

### 本番化で対応

- Supabase Auth / JWT
- 永続化
- LINE本送信
- OpenAI本接続
- 本番通知
- scheduler
- RLS SQL / local-staging verification

### 将来SaaSで対応

- 複数企業管理
- 請求
- 招待
- テナント切替
- 本格権限管理
- 運用監査

## Feedback To Codex Loop

1. feedbackをlogに記録する。
2. 個人情報や実顧客情報を除去する。
3. category / priority / severity / effort を付ける。
4. 社内確認版対応か本番化対応かを判断する。
5. 似たfeedbackをまとめる。
6. 1Loopに入れる範囲を小さく切る。
7. `docs/11_codex_tasks/` にCodex task docを作る。
8. 実装後、feedback logへ対応commitを記録する。

## Example Entries

以下は記入例です。実フィードバックではありません。

| 画面/機能 | 内容 | 種別 | 影響度 | 優先度 | 実装難易度 | 判断 |
| --- | --- | --- | --- | --- | --- | --- |
| 顧客詳細 | 「返信文の下書き」がLINE送信されるのか分かりにくい | UI / 文言 | High | P1 | S | 社内確認版で文言補強 |
| アラート | 通知タイミングを営業日だけにしたい | 通知 | Medium | P2 | L | 本番化Loopでscheduler設計 |
| 顧客情報 | 希望エリアをもっと細かく見たい | 顧客情報 | Medium | P2 | L | DB/schema計画Loopへ分離 |
| スマホ確認 | 担当者がスマホで見たい | スマホ / LIFF | Info | P4 | XL | LIFF/モバイル計画で扱う |

## Ambiguous Cases

- 文言修正に見えても、実務フローが変わる場合は `業務フロー` として扱う。
- AI下書きの語尾だけなら `AI下書き品質` かつ Effort S。本物OpenAI接続が必要なら本番化へ分ける。
- LINE送信ボタンの見た目だけなら社内確認版で対応。本物送信やaccess token利用は本番化へ分ける。
- データが消える不満は `データ永続化`。Supabase runtime接続は別Loopに分ける。
- 権限表示の違和感は `認証 / 権限`。本物JWT/session接続とは混ぜない。

## Do Not Do

- その場で全部実装しようとしない。
- 本物LINE送信など危険操作を急いでつながない。
- フィードバック内の個人情報をdocsに残さない。
- 実顧客LINE userIdを書かない。
- APIキーや `.env` 値を書かない。
- 本番ログを貼らない。
- 社内確認版と本番運用版を混同しない。
- P0/P1以外を社内確認版へ無理に詰め込まない。

## Output Of Triage

triage後は、以下のどれかに落とします。

- 次Loopで対応する小さいtask doc
- 本番化前のplanning Loop
- 将来SaaS backlog
- 対応しない判断と理由
